Supporting multiple EDS versions

EDS API is designed to allow connecting to multiple versions of EDS servers without the need to recompile any code.

To connect to a specific EDS version, please pass the appropriate version string (e.g. 9.1 or 9.2) to the constructor of a client class (eds::live::LiveClient or eds::arch::ArchClient).

In order to support a wide range of EDS versions, some precautions need to be taken when integrating EDS API into an application:

  • EDS API must always be used through the Frontend library layer.

    Do not statically link your application to any of the backend libraries.

    Do not directly call any functions exported by the backend libraries.

  • Future releases of EDS may extend API with new methods. These methods will not be available in backend libraries for older EDS versions.

    Descriptions of all eds::live::LiveClient and eds::arch::ArchClient methods contain the Since clause, which specifies the first version of EDS that implemented the method. Attempting to call a method when an older backend version is loaded will result in an eds::UnsupportedFunctionError exception being thrown.

  • Do not load different versions of the same backend at the same time (e.g. LiveClient version 9.0 and 9.1) in a single process.

    Different versions of backend libraries may internally export the same subset of symbols which could confuse the dynamic linker of the operating system.

    It is safe for a process to open connections to multiple EDS servers with the same version. That is, creating several instances of eds::live::LiveClient or eds::arch::ArchClient with the same version argument is allowed.

    However, if an application needs to simultaneously connect to several EDS servers with different versions, access to each version should be performed in a separate process.
